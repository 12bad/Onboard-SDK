#Onboard SDK 开放协议说明
---
## 本文档撰写目的
在本文档中，我们着重介绍了在开放协议中，底层指令码流的格式及配套请求本身。为了能与N1飞行控制单元进行有效通讯，工作在本层的开发者需要在程序中构建稳定且以推荐时间间隔的指令码流，通过串口（UART）传输到N1飞行控制单元，经由N1飞行控制单元对目标飞行器进行一系列操作。

**备注：本开放协议说明不包含数据透传的相关内容。**

## 协议帧
协议帧是本开放协议中信息传输的最小单位，由帧头，帧数据段及帧尾组成。以下为帧格式：

   ```
   |<--------------Header------------------>|<--Data-->|<--Tail-->|
   |SOF|LEN|VER|SESSION|ACK|RES0|PADDING|ENC|RES1|SEQ|CRC16|          DATA           |            CRC32            |
   ```

### 帧数据段

数据段类型与帧类型有关。

帧类型根据帧头段ACK标志位分为两类：0数据帧，1应答帧。
|帧类型|数据段类型|传输方向|传输内容|
|------|------------|:----------:|---------------|
|数据帧|请求数据段|机载设备<=>飞控|飞行器控制指令|
|应答帧|应答数据段|飞控<=>机载设备|控制指令的执行结果|

#### 请求数据段

```
|<-------Protocol Frame Data------->|
|CMD SET|CMD ID|CMD VALUE|
```

|字段|字节索引|大小（byte）|说明|
|----|--------|-----------------|----|
|CMD SET|0|1|请求集|
|CMD ID|1|1|请求码|
|CMD VALUE|2|与请求码有关|请求值|

### 帧尾段

帧尾段为CRC32校验值

## 协议通信机制


### 会话机制*

协议设计了会话机制，以保证请求数据和应答数据不会因为丢包而出现通信双方异常。通信双方在向对方发起通信会话时，可以根据需要通过设置在帧头部分的 SESSION 字段来选择会话方式。协议中设计了三种会话方式。

**会话方式2及方式3仅适用于具有应答数据的请求。*

|会话方式|SESSION|描述|
|--------|-------|----|
|方式1|0|发送端不需要接收端应答。|
|方式2|1|发送端需要接收端应答值，但是可以容忍应答值丢包。|
|方式3|2-31|发送端需要正确收到接收端的应答包。*|

**发送端使用这些 SESSION 发送请求数据包时，应答包中包含该请求数据包中的 帧序列号(SEQ) 和 通信过程中的会话 ID (SESSION)。如果通信过程中，发送端没有正确收到应答包，可以重新发送包含相同SESSION和SEQ的请求数据包。由于会话方式 3 是一种可靠会话方式，开发者在协议链路层实现中应考虑并实现数据丢包后的重发机制。*

## 请求集与请求码

### 请求集

**请求分为三大请求集**  

|请求集|请求集代码|说明|
|------|---------|----|
|激活验证类|0x00|查询飞控激活状态、完成激活的相关请求|
|控制请求类|0x01|控制对飞行器及云台等设备的相关请求|
|推送数据类|0x02|由飞控主动向机载设备的数据，如传感器数据、云台数据等|

### 请求码
请求集包含一系列请求码，根据具体的请求码实现相应的功能。  

请求需要在相应的权限级别下才能够被执行。当机载设备发出的请求所需的权限级别高于飞控所处权限级别时，该请求将不会被执行。较高的权限级别下可以执行低级别权限请求。

|权限级别|权限描述|
|:-----:|--------|
|0|激活请求|
|1|相机和云台的控制请求|
|2|飞行控制请求|

*备注：权限级别可通过激活请求改变，飞行器激活前默认权限级别为0。*

#### 请求码 0x01 失去控制权
失去控制权数据包会在机载控制权被夺去的时由飞控主动推送。  
机载设备的控制权优先级最低，其控制权可能在任何时候被夺去。
